# 행동 원칙

LLM의 흔한 코딩 실수를 줄이기 위한 가이드라인입니다. 반드시 따르세요.

### 1. 코딩 전에 생각하라

**추측하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현하기 전에:
- **가정**을 명시적으로 밝혀라. 확실하지 않으면 질문하라.
- 여러 해석이 가능하면 모두 제시하라 — 조용히 하나를 고르지 마라.
- 더 **단순한 접근법**이 있다면 말하라. 필요하면 반론을 제기하라.
- 뭔가 불명확하면 멈춰라. 무엇이 헷갈리는지 짚어라. 질문하라.
- **읽기 우선**: 수정 대상 코드를 반드시 먼저 읽어라. 읽지 않고 수정하지 마라.
- **환각 금지**: 존재하지 않는 API, 패키지, 파일 경로, 설정 옵션을 지어내지 마라. 확실하지 않으면 먼저 확인하라.

### 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 짐작으로 쓴 코드는 금지.**

- 요청받지 않은 기능을 추가하지 마라.
- 한 번만 쓰이는 코드에 **추상화**를 만들지 마라.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 마라.
- 발생할 수 없는 시나리오에 대한 **에러 핸들링**을 하지 마라.
- 200줄로 썼는데 50줄로 가능하면 다시 작성하라.

자문하라: "시니어 엔지니어가 이거 **오버엔지니어링**이라고 할까?" 그렇다면 단순화하라.

### 3. 외과적 변경

**건드려야 할 것만 건드려라. 본인이 남긴 부산물만 정리하라.**

기존 코드를 수정할 때:
- 주변 코드, 주석, 포매팅을 "개선"하지 마라.
- 고장나지 않은 것을 **리팩토링**하지 마라.
- 다르게 할 수 있더라도 **기존 스타일**에 맞춰라.
- 관련 없는 **데드코드**를 발견하면 언급만 하라 — 삭제하지 마라.

본인의 변경으로 사용되지 않게 된 import/변수/함수는 제거하라.

검증 기준: **변경된 모든 라인은 사용자의 요청에 직접 연결**되어야 한다.

### 4. 목표 중심 실행

**성공 기준을 정의하라. 검증될 때까지 자가점검하며 반복하라.**

작업을 **검증 가능한 목표**로 변환하라:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시켜라"
- "버그 수정" → "재현하는 테스트를 작성하고, 통과시켜라"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인하라"

다단계 작업에는 간략한 계획을 제시하라:
```
1. [단계] → 검증: [확인사항]
2. [단계] → 검증: [확인사항]
```

### 5. 점진적 실행

**대규모 변경을 한 번에 하지 마라. 작은 단위로 나눠서 각 단계마다 검증하라.**

- 여러 파일을 동시에 변경하기보다, 한 단위씩 변경하고 **중간 검증**을 수행하라.
- 뭔가 깨졌을 때 원인을 특정할 수 있도록 **변경 범위**를 최소화하라.

### 6. 실패 대응

**에러가 나면 원인부터 파악하라. 같은 시도를 반복하지 마라.**

- 에러 발생 시 증상만 고치지 말고 **근본 원인**을 분석하라.
- 같은 명령을 재시도하기 전에 왜 실패했는지 먼저 이해하라.
- 방향이 틀렸으면 고치려 하지 말고 과감히 버리고 다시 작성하라. **매몰 비용**에 집착하지 마라.

### 7. 피드백 반영

**사용자가 실수를 지적하면 기록하라. 작업 중 개선점을 발견하면 제안하라.**

- 같은 세션에서 지적받은 실수는 즉시 반영하고, 이후 동일 실수를 반복하지 마라.
- 사용자가 내 결과물을 직접 수정했다면, 수정 의도를 분석하여 컨벤션으로 제안하라.
- 코드 탐색 중 프로젝트 루트의 `context/`에 없는 도메인 지식을 발견하면 문서화를 제안하라.
- 같은 유형의 요청이 반복되면 skill 생성 또는 기존 skill 확장을 제안하라.
- 작업 흐름에서 수동/반복 작업을 발견하면 스크립트나 skill로 자동화를 제안하라.

발견 내용에 따라 적절한 위치에 반영을 제안하라:

| 발견 내용 | 반영 위치 |
|-----------|----------|
| 프로젝트 전체 규칙 | `CLAUDE.md` 또는 `.claude/rules/` |
| 도메인 구조/용어 학습 | `context/{도메인}/` |
| 특정 프로젝트 코딩 규칙 | 프로젝트 루트의 `CLAUDE.md` |

제안 형식: `[자가학습] {반영 위치}에 추가 제안: 내용 — 근거`

**임의 반영 금지**: 반드시 사용자 확인을 받은 후 반영하라.
